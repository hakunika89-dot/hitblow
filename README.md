# hitblow（チームで育てる Hit & Blow）

第2回のチーム開発のスタート地点。**最初から遊べる**：

```bash
uv sync                       # pytest など開発環境を用意
uv run python -m hitblow      # 遊ぶ
uv run pytest                 # core のテスト（緑）
```

## 構成（責務分離）

```
src/hitblow/
├─ core.py   判定と出題（純粋関数・テスト対象）
├─ game.py   進行ループ ★機能はここの「ここに足す」場所に書く
├─ cli.py    main()（第3回で hitblow コマンドが呼ぶ）
└─ __main__.py
tests/test_core.py
```

## 並行作業をスムーズにするルール（重要）

2ペア（4名なら2+2、5名なら2+3）が**別々の機能**を同時に作ります。衝突で詰まらないために：

1. **1機能 = 1ファイル**：自分の機能は**新しいファイル**に書く（例 `hint.py` / `score.py`）。
   → 機能のコードは別ファイルなので**ぶつからない**。
2. **`game.py` は自分の担当の場所だけ直す**：`game.py` には「ここに足す」場所が3か所
   （① 開始時／② 入力コマンド／③ 勝利時）。担当の場所だけに書く。
   **import も自分の場所の近くに**書く（先頭にまとめない）。
3. **このリストにだけ1行足す**（↓）。ここは2ペアが触るので**コンフリクトが起きる前提**。
   起きたら「両方残す」で直す（＝コンフリクト練習）。

### 概要

本作品は、定番の数当てゲーム「Hit & Blow」をベースに、グループのメンバーがそれぞれ独自の機能を追加して完成させたゲームです。単に機能を寄せ集めるのではなく、**「1機能=1ファイル」という設計方針**を最初に決めたことで、メンバーが同時並行で作業してもコードが衝突せず、スムーズに開発を進めることができました。

### こだわったポイント

**① 責務を分けた設計**
判定ロジック(`core.py`)と画面表示・入力(`game.py`)を分離し、テストしやすく・読みやすい構造にしました。新しいモードを追加する際も、既存コードにほとんど手を入れずに済むよう設計しています。

**② 2人対戦モード**
CPU相手だけでなく、友達同士で対戦できるモードを追加。当初は「お互いの秘密の数字を当て合う」形式でしたが、公平性を考えて「共通の1つの数字を早押しで当てる」レース形式に改良しました。さらに、対戦中に隣の人へ数字が見えてしまわないよう、入力を`*`でマスク表示する工夫も入れています。

**③ 地雷モード**
正解とは別に「地雷ナンバー」を隠し、完全一致すると即ゲームオーバーになる、スリル重視のモードです。安易に運任せにならないよう、通常のHit & Blow判定と組み合わせています。

**④ ヒント(救済)機能**
5回挑戦しても当たらないプレイヤーのために、「プライドを捨てますか？」という一度きりの選択で、正解の桁と数字を教えてくれる救済機能を追加しました。使うも使わないも自由ですが、使えるのは一度きり。対戦モードでは、ヒントの内容を相手に見られないよう画面を隠す配慮もしています。

### チーム開発として工夫したこと

- 誰がどこを触るかを事前に決め、ファイルを分けることで**コンフリクトを起こさない開発フロー**を実現
- 各機能について「これで本当に公平か」「対戦相手にバレないか」「Ctrl+Cで安全に抜けられるか」など、細部まで議論しながら改善を重ねました
- シンプルなゲームでも、遊ぶ人の立場に立って何度も仕様を見直したことで、完成度を高めています
